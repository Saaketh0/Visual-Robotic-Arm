# ROS2 Runtime Conventions

## Workspace layout

```text
xarm_ws/
├── src/
│   ├── xarm_1s_description/
│   ├── xarm_tuning_tools/
│   └── xarm_runtime/
├── build/
├── install/
├── log/
└── docs/
```

- `src/` contains source packages only.
- `build/`, `install/`, and `log/` are generated by `colcon` and must not hold hand-authored source files.

## Python package layout

Each ROS2 Python package should keep:
- `package.xml` at the package root
- `setup.py` and `setup.cfg` at the package root
- `resource/<package_name>` marker for the ament index
- runtime Python modules under `src/<package_name>/`
- `launch/`, `config/`, and `test/` at the package root

## Runtime package boundary

`xarm_runtime` is the canonical home for:
- tracker and perception-adjacent ROS2 nodes
- servo-control ROS2 nodes
- simulation and hardware backend adapters
- shared runtime helpers such as joint-model logic and backend abstractions
- the synthetic tuning publisher used to drive `pixel_error` without the camera stack

`xarm_tuning_tools` is now a compatibility package only.
`xarm_1s_description` remains the robot description and mesh asset package.

## Launch and config conventions

- Prefer package-native launch files over workspace-root script execution.
- Keep environment-specific defaults in YAML under `config/`.
- Use launch files to select sim, hardware, or tuning flows without hardcoded machine-specific paths.
- Root-level scripts should be temporary compatibility shims only.

## Jetson Nano conventions

- Assume Linux paths, not macOS-specific absolute paths, for the canonical runtime path.
- Keep device-specific defaults configurable through parameters.
- Treat serial transport, camera device selection, and inference backend settings as parameters or environment-selected options rather than embedded constants.
- Avoid Homebrew-specific or desktop-only command assumptions in runtime code when a portable alternative exists.

## Migration rules

- Migrate shared logic (for example `joint_model.py`) into `xarm_runtime` before moving full node implementations.
- Replace root-level scripts with `console_scripts` entry points after the packaged nodes reach parity.
- Delete temporary compatibility shims once launch files and tests no longer depend on them.
